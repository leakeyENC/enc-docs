# Plan: Refactor `index.html` → Next.js (App Router) docs site

## Context

`enc-docs/index.html` is a single 999-line, framework-free reimplementation of the
Encryptus documentation site. All UI is generated by string-concatenated `innerHTML`
inside one IIFE: theme tokens in a `<style>` block, eight data arrays, a `state` object
with a `go()/setState()` view switcher, and a delegated click handler. It works but is
unroutable (no real URLs, not crawlable, no SSR/SSG, no SEO metadata), and the
hover/focus behavior is bolted on via `data-hover` attributes mutating inline styles.

The goal is a modern **Next.js + TypeScript + Tailwind** port that is **fully SSG'd and
crawlable**, with real routes per guide/endpoint, SEO metadata, and React state replacing
the manual DOM mutation — while staying **visually identical** to the source (verified by
running both and diffing). `index.html` remains the source of truth and is moved to
`/reference/index.html` for visual diffing.

Two confirmed divergences from the source (user decisions):
1. **Per-route stubs**: each guide/endpoint gets its own URL + per-endpoint header &
   metadata, but the article body / API tables remain the shared onboarding example for
   now. Data and components are structured so real per-endpoint bodies can be added later
   without refactoring (e.g. optional `body`/`fields` fields on each node, falling back to
   the shared onboarding content).
2. **Theme the guide & error bodies properly**: the source hardcodes light-theme hex in
   the guide-article and error-codes bodies (they don't react to dark mode). We port those
   hex values to `--c-*` tokens so both bodies respond to the theme. Everything else
   (chrome, API view) already uses tokens and is ported verbatim.

## Stack & setup

- Scaffold Next.js (App Router) + TypeScript + Tailwind **in place** at repo root
  (`enc-docs/`). `npx create-next-app@latest . --ts --tailwind --app --src-dir --eslint`,
  package manager **npm**. The only pre-existing files are `index.html` and `.git`.
- First action: move `index.html` → `reference/index.html` (kept untouched as the diff
  baseline; add to `.gitignore`-exempt — it stays in the repo).
- `next-themes` for SSR-safe theme toggle (no flash).

## Theme tokens → globals.css + Tailwind

- Port the `.ed-app{ ... }` light token block and `.ed-app[data-theme="dark"]{ ... }`
  block **verbatim** into `src/app/globals.css`. Also port `::selection`, the
  `.ed-scroll*` scrollbar rules, `@keyframes ed-spin`/`ed-fade`, and **all** the
  `@media` blocks (1240/1080/860/560px) verbatim.
- **Theme attribute**: configure `next-themes` with `attribute="data-theme"`,
  `storageKey="ed-theme"`, `defaultTheme="light"`. next-themes writes `data-theme` on
  `<html>`. So scope the token blocks to `:root` / `:root[data-theme="dark"]` (or
  `html[data-theme="dark"]`) instead of `.ed-app`, and keep `.ed-app` as the styled
  container class on the root layout wrapper. Visually identical; makes SSR theming work.
  Preserve the existing `ed-theme` localStorage key so a returning user's choice carries.
- **Tailwind config** (`tailwind.config.ts`): map tokens to theme values reading the CSS
  vars, e.g. `colors: { accent: 'var(--c-accent)', heading: 'var(--c-heading)',
  surface: 'var(--c-surface)', border: 'var(--c-border)', ... }`. Theme switching stays
  the `data-theme` attribute — **do NOT** use Tailwind's `dark:` variant.
- **New tokens** for the few hardcoded-hex values in the guide/error bodies that have no
  existing token (per divergence #2): tip-callout (`#FFF8E1`/`#F2D98A`/`#6B5A12`),
  guide/error page bg (`#F6F8FC`), dark code-block bg (`#0B1020`/`#0B1020` error shape).
  Add light + sensible dark values. Reuse existing tokens where they already match
  (`#0A1A4F`→`--c-heading`, `#46506E`→`--c-text-secondary`, `#E2E7F1`→`--c-border-card`,
  `#EBEEF5`→`--c-border-faint`, `#FFFFFF`/`#FAFBFE`→`--c-row-1`/`--c-row-2`,
  `#7B3FE4`→`--c-type`, `#E5484D`→`--c-danger`, `#EEF1F8`/`#2B5BD7`→`--c-inline-*`).

## Fonts

- `next/font/google`: `Oswald` (400,500,600,700) + `IBM_Plex_Sans` (400,500,600,700),
  exposed as CSS variables `--font-oswald` / `--font-ibm` on the root layout.
- The source uses literal `font-family:'Oswald'` / `'IBM Plex Sans'` inline. next/font
  generates hashed family names, so replace those literals with the CSS variables
  (`font-family: var(--font-oswald)` etc.) during the JSX conversion. Set
  `--font-ibm` as the default body font on `.ed-app`. No `<link>` to Google Fonts.

## Data extraction → `src/data/`

Port these arrays **verbatim** into typed TS modules with interfaces:
- `guides.ts` — `GUIDE_NODES` (incl. nested `kyc` children); interface allows optional
  per-node body/metadata for the stub structure.
- `api.ts` — `API_GROUPS` (`id, items[]{id,m,path,label,desc}`), plus `endpointById()`
  helper and the shared `ONBOARDING_FIELDS`, `RESP_FIELDS`, `EP_ERR_FIELDS`,
  `REQUEST_JSON`, `RESPONSE_JSON`, `CURL_DEFAULT`, and the `tsSnippet()` generator.
- `errors.ts` — `ERROR_GROUPS`, `errShape`.
- `site.ts` — `SOCIALS`, `FOOTER_COLS`, popular-topics list.
Method type: `type Method = 'GET'|'POST'|'PUT'|'PATCH'|'DELETE'`.

## Routing & rendering

Replace `go()/setState` view switching with real routes (all SSG via
`generateStaticParams`, each with `generateMetadata` for title+description):

| Route | Source view | Notes |
|---|---|---|
| `/` | `viewHome()` | Home (hero, cards, popular topics, CTA banner) |
| `/guides/[slug]` | `viewGuide()` | slugs from `GUIDE_NODES` + nested KYC children |
| `/api/[group]/[endpoint]` | `viewApi()` | params from `API_GROUPS` (group id slugified) |
| `/errors` | `viewErrors()` | error codes reference |

- `/guides` and `/api` index entry points redirect to first child (matches source: nav
  "Guides" → `overview`, "API Reference" → first endpoint `partners-onboarding`).
- Active-state highlighting in sidebars derives from the current route params (via
  `useParams`/`usePathname` in the client sidebar), replacing
  `state.activeGuide/activeEndpoint`.
- Collapsible sidebar groups (`openGuide`/`openApi`): default-open the group containing
  the active route; toggle is local client state in the sidebar component.

## Components (Server by default; `"use client"` only where noted)

Layout (`src/app/layout.tsx` + `src/components/`):
- `ThemeProvider` (client wrapper around next-themes) in root layout; `.ed-app` class +
  font vars on the body/wrapper.
- `Header` — server shell; contains client islands: `ThemeToggle`, `SearchTrigger`
  (opens overlay), `MobileMenuButton`. Nav pills + "Get API Key" CTA are plain links.
- `Footer` — server (socials, `FOOTER_COLS`, status line).
- `GuideSidebar` / `ApiSidebar` — **client** (collapsible groups + active state).
- `TableOfContents` — server (static anchor links per page).

Pages / sections:
- `Home` — server.
- `GuideArticle` — server; renders shared onboarding body (themed per divergence #2),
  prev/next nav, "was this helpful" widget.
- `ApiReference` — server shell rendering per-endpoint header (method badge, path, label,
  desc) + shared request/response/error tables + example JSON blocks; embeds the client
  `TryItConsole` in the right rail.
- `ErrorCodes` — server.

Shared:
- `SearchOverlay` — **client** (Cmd/Ctrl-K, ESC, input filtering, keyboard nav, navigates
  via `next/link`/`router.push`). Builds the same search index from data.
- `MobileDrawer` — **client** (open/close, theme row, nav links).
- `MethodBadge` — server (soft + solid variants, token-driven by method).
- `DataTable` — server (the `grid-template-columns` request/response/error tables).
- `CodeBlock` — server (`<pre>` with `ed-scroll`, monospace, token colors).

Conversion rules:
- All `innerHTML` string concat → JSX. Inline `style="..."` strings → JSX `style={{}}`
  objects or Tailwind classes (keep token references via `var(--c-*)`).
- Replace the `data-hover` / `data-focus` / `applyDynamicStyles` pattern with real CSS
  `:hover` / `:focus` classes in `globals.css` (one utility class per distinct effect,
  e.g. `.ed-card-hover`, `.ed-navrow-hover`, `.ed-toc-link`, `.ed-footer-link`).

## Try It console (`TryItConsole`, client)

Flagship upgrade from the static fake. Replace the `setTimeout(1150)`-then-show-static-JSON
behavior with a real interactive island:
- cURL / TypeScript tab toggle; editable `<textarea>` bound to React state (seeded from
  `CURL_DEFAULT` / `tsSnippet(ae)`), Copy button (clipboard + "Copied!" flash).
- Auth-token input + Send button that fires a real `fetch` to
  `${process.env.NEXT_PUBLIC_API_BASE_URL}${endpoint.path}` with the chosen method and
  `Authorization: Bearer <token>`.
- Response viewer: loading spinner (`ed-spin`), status line (`201 Created · NNms`),
  syntax-highlighted JSON. **If `NEXT_PUBLIC_API_BASE_URL` is unset**, fall back to the
  stubbed `RESPONSE_JSON` after a short delay (preserves current demo behavior).
- `.env.local.example` documents `NEXT_PUBLIC_API_BASE_URL`.
